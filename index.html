Class · Object · Constructor
Design a class BankAccount with attributes accountNumber, holderName, and balance. Implement a parameterized constructor, a deposit() method, a withdraw() method (with insufficient-funds check), and a display() method. Create two objects and demonstrate all operations.
15 pts

class BankAccount {
private:
    int    accountNumber;
    string holderName;
    double balance;
public:
    // 1. Parameterized constructor
    BankAccount(int accNo, string name, double bal) { /* ... */ }

    // 2. deposit(amount)
    void deposit(double amount) { /* ... */ }

    // 3. withdraw(amount) – check insufficient funds
    void withdraw(double amount) { /* ... */ }

    // 4. display account info
    void display() { /* ... */ }
};

Model Answer

#include<iostream>
#include<string>
using namespace std;

class BankAccount {
private:
    int    accountNumber;
    string holderName;
    double balance;
public:
    BankAccount(int a, string n, double b)
        : accountNumber(a), holderName(n), balance(b) {}

    void deposit(double amt) {
        if(amt > 0) balance += amt;
    }

    void withdraw(double amt) {
        if(amt > balance) cout << "Insufficient funds!\n";
        else balance -= amt;
    }

    void display() {
        cout << "Acc: " << accountNumber << " | Name: "
             << holderName << " | Balance: " << balance << endl;
    }
};

int main() {
    BankAccount a1(1001, "Ali", 5000.0);
    BankAccount a2(1002, "Sara", 3000.0);
    a1.deposit(2000); a1.display();
    a2.withdraw(5000); // triggers insufficient funds
    return 0;
}

Function Overloading · Polymorphism

OOP Limitations · Design Tradeoffs
Write a program that illustrates a common disadvantage of OOP — tight coupling through deep inheritance. Create 4 levels of inheritance: Vehicle → Car → ElectricCar → TeslaModel3. Show how a change in the base class Vehicle (e.g., adding a required method) forces changes across all derived classes. Comment your code to explain the problem.
15 pts

Model Answer + Explanation

// Deep inheritance → tight coupling
class Vehicle {
public:
    virtual void fuelType() = 0; // adding this forces ALL derived to implement
    virtual void move() { cout << "Moving\n"; }
};

class Car : public Vehicle {
public:
    void fuelType() { cout << "Petrol\n"; }
};

class ElectricCar : public Car {
public:
    void fuelType() { cout << "Electric\n"; } // must override
};

class TeslaModel3 : public ElectricCar {
public:
    void fuelType() { cout << "Battery 100kWh\n"; }
};
// Disadvantage: Every level must implement fuelType()
// → brittle design, high maintenance cost

All Concepts Combined
Build a University System with: (a) base class Person (encapsulation), (b) derived classes Student and Teacher (inheritance), (c) overloaded constructor in each, (d) a pure virtual method getRole() (polymorphism), (e) an overloaded + operator in Student to merge two students' GPAs (function overloading). Model Answer

class Person {
protected: string name; int age;
public:
    Person(string n, int a): name(n), age(a){}
    virtual string getRole() = 0;
};

class Student : public Person {
    double gpa;
public:
    Student(string n, int a, double g): Person(n,a), gpa(g){}

    // operator overloading – average GPA

};

T1

Class vs Object

Theory Points
01
Class · Object · OOP Basics
Define a class and an object in OOP. What is the difference between them? Give a real-world analogy and a code-level example to support your answer.
5 pts

Model Answer

  • Class: A blueprint or template that defines attributes (data) and methods (behavior) for a category of objects. It does not occupy memory by itself.
  • Object: A runtime instance of a class. It occupies memory and holds actual values.
  • Analogy: Class = "Car blueprint"; Object = "Your specific red Toyota Corolla".
  • Code: class Car {…}; is the class; Car myCar; is the object.
  • Multiple objects can be created from one class, each with different state.
T2

Encapsulation

Theory Points
02
Encapsulation · Data Hiding
What is encapsulation? Why is it considered one of the most important pillars of OOP? Explain data hiding and how access specifiers (private, protected, public) enforce it.
5 pts

Model Answer

  • Encapsulation: Bundling data (attributes) and methods that operate on that data into a single unit (class), while restricting direct access from outside.
  • Data hiding: Making attributes private so they cannot be directly modified by outside code — changes only happen through controlled methods (getters/setters).
  • private → accessible only within the class.
  • protected → accessible within the class and its derived classes.
  • public → accessible from anywhere.
  • Benefit: Prevents invalid state, improves maintainability, hides implementation details.
T3

Inheritance

Theory Points
03
Inheritance · Types
Define inheritance and list its types (single, multiple, multilevel, hierarchical, hybrid). What is the difference between method overriding and method overloading? Why do we use the virtual keyword in C++?
5 pts

Model Answer

  • Inheritance: A mechanism where a derived class acquires properties and behaviors of a base class, enabling code reuse.
  • Types: Single (A→B), Multiple (A+B→C), Multilevel (A→B→C), Hierarchical (A→B, A→C), Hybrid (combination).
  • Overriding: Same method signature in derived class replaces the base version (runtime polymorphism).
  • Overloading: Same method name but different parameters in the same class (compile-time).
  • virtual keyword: Tells the compiler to use the derived class's version of the method when called via a base pointer — enables dynamic dispatch.
T4

Benefits of OOP

Theory Points
04
OOP Advantages
List and explain any five benefits of Object-Oriented Programming compared to procedural programming. Provide a brief real-world scenario where each benefit is clearly visible.
5 pts

Model Answer

  • Reusability: Inheritance allows reusing existing code. E.g., Employee reuses all logic from Person.
  • Modularity: Each class is an independent module — easy to develop, test, and debug separately.
  • Maintainability: Encapsulation localizes changes. Fixing a bug inside a class doesn't affect code elsewhere.
  • Flexibility (Polymorphism): One interface, many implementations. Adding new types doesn't break existing code.
  • Real-world modelling: Objects map naturally to real entities (BankAccount, Car, Student), making design intuitive.
T5

Disadvantages of OOP

Theory Points
05
OOP Limitations
Mention and explain five disadvantages or limitations of OOP. When is procedural or functional programming a better choice than OOP?
5 pts

Model Answer

  • Steeper learning curve: Concepts like polymorphism, virtual tables, and memory management are harder for beginners.
  • Overhead: Objects consume more memory; virtual function calls are slower than direct calls due to vtable lookup.
  • Not suitable for small tasks: Writing a class for a simple script is unnecessary complexity.
  • Tight coupling in deep inheritance: Changes in a base class can break all derived classes (Fragile Base Class problem).
  • Slower development start: Designing a good class hierarchy takes time upfront.
  • Better alternatives: For data pipelines and mathematics → functional; for system utilities/scripts → procedural.
